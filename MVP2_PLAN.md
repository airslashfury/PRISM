# PRISM MVP2 Plan — "Make the consequences visible"

> North star (unchanged): the objective is not to make decisions — it is to make the
> *consequences* of decisions easy to see. MVP1 built the engine and a dashboard.
> MVP2 makes PRISM feel like a living model of the island: terrain you can read,
> scenarios you can play, answers you can ask for, and a backend that won't fall over
> when we add weight.

This plan supersedes/absorbs the remaining FRONTEND_PLAN.md phases (F4 Decision+Auth,
F5 HA hardening) — they are folded into M3 and M6 below.

## Phase map

| Phase | Name | Sessions (est.) | Depends on | Build model |
|---|---|---|---|---|
| M1 | AI text quality + delegation hygiene | 1 | — | Sonnet (Haiku for copy passes) |
| M2 | 3D Corridor Experience | 2–3 | M1 (NarrativePanel) | Sonnet; Opus on draping architecture decision |
| M3 | Backend stability spine | 2–3 | — (parallel with M2) | Sonnet; Opus on HA architecture review |
| M4 | Playground | 3–4 | M3 (job queue), M2 (3D + draw patterns) | Opus design, Sonnet build, Haiku boilerplate |
| M5 | Signature features | 1–2 each, independently shippable | M3 (M5b needs M4 graph overlay) | Sonnet; Opus for tool-use agent design |
| M6 | Auth + multi-user + K8s (elective) | 1–2 | M3 | Sonnet |

**Gate protocol (unchanged):** every phase "Done when" → hand off to `phase-gate-reviewer`
(Opus) for GO/NO-GO before the next phase. Bulk passes → `bulk-classifier` (Haiku).
Main session stays on Sonnet; design-heavy sessions (M4 kickoff, M5c agent design) may run
on Opus/Fable by explicit `/model` choice.

## M1 — AI text quality + model delegation hygiene

**Goal:** AI text everywhere in the product reads like a publication, not a model dump.
Closes the Phase 7 "empty narrative" carry-forward. Cheapest-capable-model delegation
formalized for all new task types.

**Why first:** one session, touches every later phase (M2 narrative panel, M4 scenario
summaries, M5 Ask-PRISM answers all reuse this), and fixes the most visible quality bug.

### Build tasks

1. **Frontend markdown rendering** — add `react-markdown` + `remark-gfm` +
   `@tailwindcss/typography`. New shared component `frontend/components/narrative-panel.tsx`:
   renders narrative markdown with the dashboard's dark `prose` styling, a "generated by
   {tier} · {timestamp}" footer, and a skeleton/streaming state. Replace the raw
   `{detail.narrative}` div in `corridor/page.tsx` and use it on every page that shows
   AI text from here on. **No page ever interpolates LLM output as a bare string again.**
2. **Output contract in the backend** — `prism/report/narrative.py`: every narrative call
   gets a system prompt block (cached) that pins the format: GitHub-flavored markdown,
   `###` section headers from a fixed list, no preamble ("Here is…" banned), no code
   fences, lead with the consequence per the ui-ux skill. Add `format: "markdown"` to the
   stored JSON.
3. **Validation before insert** — reject/retry empty or sub-200-char completions before
   persisting to `report.narratives` (closes the Phase 7 carry-forward: 2 of 3 early
   narratives were empty). One retry at the same tier, then one escalated tier, then store
   an explicit stub with `status: "failed"` — never silent empties.
4. **Streaming endpoint** — `POST /reports/narratives/stream` (SSE) so the UI shows text
   as it generates instead of a long spinner. `prism/llm.py` grows a `stream=True` path
   for the Anthropic backend (Ollama falls back to non-streaming).
5. **models.yml hygiene** — fix the `claude-opus-4-x` placeholder (→ `claude-opus-4-8`),
   add routing for all MVP2 task types (see "Model delegation" section at the bottom),
   keep auto-escalation thresholds as-is.
6. **ui-ux skill update** — add the narrative-rendering rule and (placeholder) Playground
   page guidance to `.claude/skills/ui-ux/SKILL.md` so future copy passes inherit it.

**Done when:** corridor narrative renders as styled markdown; a deliberately empty LLM
response does not reach the DB; SSE endpoint streams a narrative end-to-end;
`config/models.yml` has no placeholder IDs; `pytest` green.

## M2 — 3D Corridor Experience

**Goal:** topography reads instantly (you can *see* why Alt 3 costs $13.1B), and rail
alternatives look like engineered alignments, not GPS traces.

### The core problem to fix first

Deck.gl layers do **not** drape on MapLibre terrain. Today's "3D" toggle raises the
basemap into relief, but route lines stay at z=0 and slice through mountains. Two fixes,
**do (a), keep (b) as fallback** — decide via a short Opus design review at phase start:

- **(a) Pre-computed 3D alignments (recommended):** new API endpoint
  `GET /corridor/routes/{id}/profile` samples the mirrored 3DEP DEM along the route at
  ~100 m intervals → returns `[lng, lat, elev_m]` plus grade %. Render with `PathLayer`
  using true `[x, y, z]` coordinates (+ a small vertical offset so the ribbon sits on the
  surface). Works with the existing MapLibre terrain, no renderer change, and the same
  data powers the elevation-profile chart below. DEM sampling lives in
  `prism/terrain/profile.py` (rasterio, reads the raw GeoTIFFs — no new DB tables needed).
- **(b) Deck.gl `TerrainLayer` + `_TerrainExtension`:** moves terrain into deck so all
  layers drape natively. More invasive (replaces MapLibre terrain + hillshade); only if
  (a)'s visual result disappoints.

### Build tasks

1. **Elevation profile API + chart** — `/corridor/routes/{id}/profile` (above); new
   `frontend/components/charts/elevation-profile.tsx`: longitudinal section (distance vs
   elevation), area fill, terrain-type bands (standard/elevated/tunnel colored along the
   x-axis), grade coloring where >3.5%. This is the single most "real rail planning"
   artifact we can add — every rail study has one.
2. **True-3D route ribbons** — `PathLayer` with z-coords, `widthUnits: "meters"`
   (~250 m), rounded joints. Terrain-typed treatment: **elevated** segments render at
   `z + 25 m` with `ColumnLayer` viaduct piers every ~400 m; **tunnels** render as a
   dimmed dashed path at surface + portal icon markers at both ends (don't try to render
   underground — portals communicate it better).
3. **Topography presentation** — exaggeration slider (1×–3×, default 1.7); satellite
   basemap toggle (Esri World Imagery, free with attribution) alongside dark-matter;
   MapLibre sky/atmosphere + fog when pitched; optional contour overlay from the DEM
   tiles at high zoom.
4. **Animated train** — `TripsLayer` running a marker along the selected alignment at a
   real cruise speed (~110 km/h scaled), with play/pause. Cheap, sells "this is a train."
5. **Corridor fly-through** — "Tour" button: camera animates along the route at pitch
   ~60° (chain of `flyTo` keyframes from the profile points), narration panel highlights
   the segment being flown (terrain type, cost/km).
6. **Stations** — render endpoint + intermediate station markers (`IconLayer`) from the
   corridor endpoints and the SERVES-linked barrios; closes the Phase 10 "station
   entities" carry-forward at the visual level (graph entities can follow in M4).

**Done when:** with 3D on, no route line clips through terrain; elevation profile renders
for all 5 alternatives and its total length matches `corridor.routes.total_km` within 2%;
tunnel/elevated segments are visually distinct in 3D; train animation and fly-through run
at 60 fps on the dev box; typecheck + lint clean.

## M3 — Backend stability spine

**Goal:** the app stops doing heavy work in request handlers, survives restarts, and is
ready for the load M4/M5 add. Absorbs FRONTEND_PLAN F5.

### Build tasks

1. **Redis** — new `redis:7-alpine` service in compose (with volume + healthcheck). Three
   consumers, in priority order:
   - **API response cache:** decorator on heavy endpoints (flood GeoJSON 1.88 MB,
     transmission 1.85 MB, tract choropleth) — keyed by route+params, TTL aligned to
     `sync_interval_hours`; **invalidated by the sync spine** (a sync that updates a layer
     deletes its cache keys — wire into `prism/sync/resync.py`).
   - **Job queue:** `arq` (async, fits FastAPI) + a `worker` container running corridor
     generation, playground evaluation (M4), rescore triggers, narrative generation.
     Job status endpoints `GET /jobs/{id}`; SSE progress channel via Redis pub/sub.
   - **Rate limiting:** `slowapi` with Redis storage on mutation + LLM endpoints.
2. **Vector tiles for heavy layers** — `ST_AsMVT` tile endpoints
   (`/tiles/{layer}/{z}/{x}/{y}.mvt`) for flood zones, transmission, tracts; Deck.gl
   `MVTLayer` replaces the 1.9 MB GeoJSON fetches. Tiles cached in Redis. This is the
   single biggest perceived-performance win for the map pages and is mandatory before
   Playground draws on top of these layers.
3. **Migrations** — Alembic baseline capturing the current schema; all future DDL
   (playground schema in M4) ships as migrations, not ad-hoc `schema.py` DDL. Existing
   `schema.py` files stay as the source the baseline is generated from.
4. **Postgres durability** — SQLAlchemy pool tuning (pool_pre_ping, sane max);
   nightly `pg_dump` to `data/backups/` via a compose cron sidecar (or Windows Task
   Scheduler entry, documented); restore drill documented in `docs/runbook.md`.
5. **Production posture** — `docker-compose.prod.yml`: gunicorn+uvicorn workers for the
   API, nginx (or Caddy) in front, restart policies, resource limits, healthchecks on all
   services; structured JSON logs; `/metrics` (Prometheus) on api + worker.
6. **CI** — GitHub Actions: ruff + mypy + pytest (against a PostGIS service container,
   markers to skip data-heavy tests), eslint + tsc + `next build`, docker image builds.
7. **Registry hygiene** — clean `_test_*` rows from `sync.data_sources` via a dedicated
   test schema/teardown (closes a Phase 9 carry-forward).

**Done when:** `docker compose -f docker-compose.prod.yml up` runs postgis + redis + api +
worker + frontend + nginx cleanly; flood/transmission render via MVT with first-tile
latency < 300 ms warm; a queued corridor-generation job survives an API container restart;
CI green on a fresh clone; one documented backup + restore executed successfully.

## M4 — Playground

**Goal:** a sandbox where a planner sketches *any* infrastructure — substation, line,
road, bridge, rail, water — onto the live model and immediately sees cost, capacity,
failure impact, and objective value. This is the payoff of the pluggable-asset
architecture: the palette is literally `prism/assets/*`.

### Design rules (decide once, Opus review at phase start)

- **Playground never mutates base tables.** Scenarios are copy-on-write overlays: a
  scenario = base graph + a set of additions/removals/failures applied at evaluation time.
- **Evaluation is always a job** (arq, from M3) — never inline in a request handler.
- **Every asset type that implements `InfrastructureAsset` appears in the palette
  automatically** via an asset-registry endpoint; adding a future asset class requires
  zero frontend changes.

### Build tasks

1. **Schema** (Alembic migration) — `playground.scenarios` (id, name, description,
   author, status, created/updated), `playground.scenario_assets` (scenario_id,
   asset_type, geom 32161, params jsonb, op = add|remove), `playground.scenario_events`
   (failure injections: entity_id, event_type), `playground.scenario_results`
   (scenario_id, run_id, objective breakdown jsonb, resilience_delta jsonb, computed_at).
2. **Asset registry API** — `GET /playground/asset-types`: reflects over
   `prism/assets/*` → type, geometry kind (point/line), parameter schema (e.g. rail:
   standard/elevated/tunnel; road: lanes; substation: capacity), default unit costs,
   icon hint.
3. **Evaluation engine** — `prism/playground/evaluate.py`: for each draft asset, run the
   four models (construction, maintenance NPV, capacity, failure impact) +
   `objective_value()`; for linear assets, segment against the M2 cost surface (flood
   premium, terrain type inference from slope). Aggregate to a scenario scorecard. For
   scenarios with failure injections or grid changes: rebuild the affected subgraph as a
   NetworkX overlay and re-run cascade/composite scoring for touched substations →
   resilience delta vs baseline.
4. **What-if failure mode** — click any *existing* entity (substation, bridge) →
   "Fail it" → downstream traversal via `graph.relationships` (Phase 2 `downstream_of()`)
   → map ripple: barrios/hospitals/water plants losing service, with population counts.
   This is the demo that lands with non-technical stakeholders.
5. **Frontend `/playground`** — draw tools (`@deck.gl-community/editable-layers`):
   place points, draw lines with vertex editing; **auto-route assist** for linear assets
   (snap a rough sketch to the M2/Phase-10 cost surface via the Dijkstra router — "draw
   roughly, PRISM refines"); parameter side panel; Evaluate button → job progress (SSE) →
   scorecard (objective breakdown per the standard formula, resilience delta, headline
   consequence line per ui-ux skill); scenario list with save/duplicate/delete.
6. **Scenario comparison** — pick two scenarios → side-by-side delta cards + map swipe
   divider; AI comparison narrative (Sonnet via `tradeoff_comparison` route, rendered
   through the M1 NarrativePanel).
7. **Station entities** — when a rail scenario is committed as "reference", create
   station nodes in `graph.entities` + SERVES edges (closes the Phase 10 carry-forward
   properly).

**Done when:** a user can draw a transmission line + a substation, evaluate, and get a
scorecard with all four model outputs and objective value; failing PALO SECO SP TC shows
its real downstream footprint on the map; two scenarios compare side-by-side with an AI
narrative; base tables provably untouched (test: checksum base tables before/after a
playground session); 25+ new tests green.

## M5 — Signature features (the shine)

Each independently shippable; recommended order below. All AI text flows through M1's
contract; all heavy work through M3's queue.

### M5a — Consequence Lens (1 session)
Hover/long-press any asset on any map → instant ripple-highlight of its downstream
dependency cone (FEEDS/POWERS/SERVES traversal, pre-computed per entity into a cached
lookup) with a one-line consequence: "Failure cuts power to 88K people, 2 hospitals,
1 water plant." The north star as a UI primitive. Backend: materialized
`graph.downstream_summary` table refreshed by the sync spine.

### M5b — Ask PRISM (1–2 sessions)
Natural-language query bar in the topbar. Claude tool-use agent (Anthropic SDK) over a
small set of typed tools that wrap existing API/queries: `find_entity`, `downstream_of`,
`top_resilience`, `portfolio_items`, `corridor_compare`, `svi_lookup`. Haiku parses/routes
(`query_routing`), Sonnet composes the answer; answers cite live numbers and **drive the
map** (the response includes entity ids → frontend highlights them). Guardrail: read-only
tools, no SQL passthrough. "Which barrios lose water if Palo Seco fails?" becomes a
10-second interaction instead of a CLI session.

### M5c — Storm Timeline (2 sessions)
Animated hurricane scenario: a parametric Cat-3 track sweeps the island on a time
scrubber; substations fail as the wind field crosses their hazard envelope (Phase 3
scores), cascades propagate per timestep, the consequence counter (people/hospitals
without power) climbs in real time. Precomputed timeline (a job) → `TripsLayer` +
timestamped failure events. Extends the rescore trigger beyond cat3-only as a side
effect (closes that Phase 9 carry-forward).

### M5d — Report Studio (1 session)
One-click board pack: pick a portfolio run / corridor / playground scenario → server-side
PDF (WeasyPrint or Playwright print-to-PDF of a print-styled route) bundling the maps
(static renders), tables, objective breakdowns, and a flagship Opus narrative
(`flagship_report` route). The artifact a planner forwards to a government partner.

**Done when (per feature):** demo-able end-to-end on live data, no console errors, gate
review per feature batch (a/b together is fine).

## Sequencing & risk notes

- **M2 and M3 can run in parallel sessions**; M4 must wait for M3's queue and ideally
  M2's draw/3D patterns.
- **Biggest technical risk:** deck.gl-on-terrain visual quality (M2). Mitigation: the
  profile-endpoint approach (a) is useful regardless — the elevation chart ships even if
  ribbons need iteration.
- **Biggest scope risk:** Playground. Hold the line on copy-on-write overlays and
  job-based evaluation; resist live-mutation "quick wins."
- **Data carry-forwards that still bite:** CRIM parcels (property impact in Playground
  scorecards uses the Census proxy — label it as such in the UI), bridge spans (default
  tier), ~50% `eid=` names (pre-resolve names into `graph.downstream_summary` during M5a).
